# REST API

---

# Why

Modern applications consist of a frontend and a backend.

The frontend needs a standard way to communicate with the backend.

REST API provides a simple and organized way for clients and servers to exchange data over HTTP.

---

# What

REST (Representational State Transfer) is an architectural style for designing web APIs.

A REST API exposes resources through URLs and uses HTTP methods to perform operations on those resources.

Example

```text
/users
/blogs
/comments
```

Each URL represents a resource.

---

# REST Principles

A REST API follows these principles:

- Client-Server Architecture
- Stateless Communication
- Resource-Based URLs
- Standard HTTP Methods
- Uniform Interface

---

# Resources

Everything in a REST API is treated as a resource.

Examples

```text
/users

/blogs

/comments

/categories
```

Each resource has its own endpoint.

---

# REST Endpoints

Get all blogs

```http
GET /blogs
```

Get one blog

```http
GET /blogs/:id
```

Create blog

```http
POST /blogs
```

Update blog

```http
PATCH /blogs/:id
```

Delete blog

```http
DELETE /blogs/:id
```

---

# Stateless

REST APIs are stateless.

This means:

Every request must contain all the information required to process it.

The server does not remember previous requests.

Authentication is usually handled using JWT tokens sent with every request.

# REST API Design

---

# Why

A REST API should be easy to understand, easy to use, and consistent.

Good API design makes it easier for:

- Frontend developers
- Backend developers
- Team members
- Future maintenance

A poorly designed API causes confusion and makes applications harder to develop.

---

# What

REST API Design is the process of organizing API endpoints, HTTP methods, request bodies, responses, and URLs in a clean and consistent way.

The goal is to make APIs predictable and easy to understand.

---

# Use Nouns Instead of Verbs

Endpoints should represent **resources**, not actions.

✅ Good

```text
/users
/blogs
/comments
/categories
```

❌ Bad

```text
/getUsers
/createBlog
/deleteComment
```

The HTTP method already describes the action.

---

# Use Correct HTTP Methods

| Action | Method |
|---------|--------|
| Create | POST |
| Read | GET |
| Update | PATCH |
| Replace | PUT |
| Delete | DELETE |

Example

```http
POST /blogs
```

Creates a blog.

```http
GET /blogs
```

Returns all blogs.

```http
PATCH /blogs/:id
```

Updates a blog.

```http
DELETE /blogs/:id
```

Deletes a blog.

---

# Use Plural Resource Names

Use plural nouns for resources.

✅ Good

```text
/users
/blogs
/comments
```

❌ Bad

```text
/user
/blog
/comment
```

---

# Use Route Parameters

When accessing a specific resource, use route parameters.

Example

```http
GET /blogs/:id
```

```http
PATCH /blogs/:id
```

```http
DELETE /blogs/:id
```

---

# Use Query Parameters

Use query parameters for:

- Searching
- Filtering
- Pagination
- Sorting

Examples

```http
GET /blogs?page=1
```

```http
GET /blogs?category=Technology
```

```http
GET /blogs?sort=latest
```

---

# Keep URLs Simple

Good

```text
/users

/blogs

/blogs/:id
```

Avoid unnecessary nesting.

Bad

```text
/getAllBlogsForEveryUser
```

---

# Use Consistent Responses

Success Response

```json
{
    "message": "Blogs fetched successfully",
    "data": []
}
```

Error Response

```json
{
    "message": "Blog not found"
}
```

Keep the response structure consistent throughout the project.

---

# Use Proper Status Codes

Examples

```text
200 OK

201 Created

400 Bad Request

401 Unauthorized

403 Forbidden

404 Not Found

409 Conflict

500 Internal Server Error
```

---

# API Versioning

As APIs evolve, create versions.

Example

```text
/api/v1/users

/api/v1/blogs
```

If major changes are introduced later:

```text
/api/v2/users
```

This prevents breaking older applications.

# Common Mistakes

### Mistake 1

Using verbs in URLs.

Wrong

```text
/createBlog
```

Correct

```text
POST /blogs
```

---

### Mistake 2

Using the wrong HTTP method.

Wrong

```text
GET /deleteBlog/123
```

Correct

```text
DELETE /blogs/123
```

---

### Mistake 3

Returning different response formats.

Wrong

```json
{
    "user": {}
}
```

Another API

```json
{
    "result": {}
}
```

Use one consistent response structure.

---

### Mistake 4

Ignoring API versioning.

Always include a version for production APIs.

Example

```text
/api/v1/
```

---

# Best Practices

- Design APIs around resources.
- Use nouns in URLs.
- Follow standard HTTP methods.
- Keep endpoints short and meaningful.
- Return consistent JSON responses.
- Use proper HTTP status codes.
- Version your APIs.

---

# Summary

- REST API Design focuses on creating clean, predictable, and maintainable APIs.
- Use resource-based URLs.
- Use the correct HTTP methods.
- Keep response formats consistent.
- Use route parameters and query parameters appropriately.
- Follow REST conventions to make APIs easier to understand and maintain.
