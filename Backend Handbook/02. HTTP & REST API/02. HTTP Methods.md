# Status Codes

---

# Why

When a client sends a request to the server, the server must tell the client whether the request was successful or not.

HTTP Status Codes are standardized numbers that describe the result of a request.

They help the client understand what happened without reading the entire response.

---

# What

An HTTP Status Code is a **three-digit number** returned by the server with every HTTP response.

Example

```http
HTTP/1.1 200 OK
```

Here,

- `200` is the status code.
- `OK` is the status message.

---

# Status Code Categories

| Range | Meaning |
|--------|---------|
| 1xx | Informational |
| 2xx | Success |
| 3xx | Redirection |
| 4xx | Client Error |
| 5xx | Server Error |

In backend development, developers mostly work with:

- 2xx
- 4xx
- 5xx

---

# 200 OK

The request was successful.

Use when:

- Fetching data
- Updating data
- Deleting data
- Login successful

Example

```js
return res.status(200).json({
    message: "Blogs fetched successfully",
    data: blogs
});
```

---

# 201 Created

The server successfully created a new resource.

Use when:

- Register User
- Create Blog
- Upload Image

Example

```js
return res.status(201).json({
    message: "User created successfully",
    data: user
});
```

---

# 400 Bad Request

The client sent an invalid request.

Use when:

- Required fields are missing
- Invalid request body
- Validation fails

Example

```js
if (!title || !description) {

    return res.status(400).json({
        message: "All fields are required"
    });

}
```

---

# 401 Unauthorized

The client is not authenticated.

Use when:

- JWT Token is missing
- Invalid Token
- User is not logged in

Example

```js
return res.status(401).json({
    message: "Unauthorized"
});
```

---

# 403 Forbidden

The client is authenticated but does not have permission.

Use when:

- User tries to edit another user's blog
- User tries to access restricted resources

Example

```js
return res.status(403).json({
    message: "You are not authorized to perform this action"
});
```

---

# 404 Not Found

The requested resource does not exist.

Use when:

- Blog not found
- User not found
- Invalid Route

Example

```js
return res.status(404).json({
    message: "Blog not found"
});
```

---

# 409 Conflict

The requested operation conflicts with existing data.

Use when:

- Email already exists
- Username already exists

Example

```js
return res.status(409).json({
    message: "Email already registered"
});
```

---

# 500 Internal Server Error

Something went wrong on the server.

Usually caused by:

- Database errors
- Unexpected exceptions
- Server crashes

Example

```js
return res.status(500).json({
    message: "Internal Server Error"
});
```

---

# Common Status Codes Used in Backend

| Status Code | Meaning | Common Usage |
|-------------|---------|--------------|
| 200 | OK | Fetch, Update, Delete |
| 201 | Created | Create Resource |
| 400 | Bad Request | Invalid Input |
| 401 | Unauthorized | Login Required |
| 403 | Forbidden | Permission Denied |
| 404 | Not Found | Resource Doesn't Exist |
| 409 | Conflict | Duplicate Data |
| 500 | Internal Server Error | Server Failure |

# Choosing the Correct Status Code

| Situation | Status Code |
|-----------|-------------|
| Get Blogs Successfully | 200 |
| Register User Successfully | 201 |
| Missing Required Fields | 400 |
| Invalid JWT Token | 401 |
| User Editing Another User's Blog | 403 |
| Blog Doesn't Exist | 404 |
| Email Already Exists | 409 |
| Unexpected Server Error | 500 |

# Common Mistakes

### Mistake 1

Returning `200` for every response.

Wrong

```js
return res.status(200).json({
    message: "Email already exists"
});
```

Correct

```js
return res.status(409).json({
    message: "Email already exists"
});
```

---

### Mistake 2

Using `500` for validation errors.

Validation errors are client mistakes.

Use:

```text
400 Bad Request
```

---

### Mistake 3

Confusing `401` and `403`.

**401**

User is **not authenticated**.

**403**

User is authenticated but **not allowed** to perform the action.

---

# Best Practices

- Always return meaningful status codes.
- Keep response messages clear and consistent.
- Follow standard HTTP conventions.
- Avoid using `200 OK` for failed requests.
- Choose the status code that best describes the result of the request.

---

# Summary

- Every HTTP response includes a status code.
- Status codes tell the client whether the request succeeded or failed.
- `200` and `201` indicate success.
- `400`, `401`, `403`, `404`, and `409` indicate client-related errors.
- `500` indicates a server-side error.
- Using correct status codes makes APIs easier to understand, debug, and maintain.
