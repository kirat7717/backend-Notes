# Controllers

---

# Why

A route should only decide **which function should handle the request**.

If we write all business logic inside routes, the code becomes:

- Difficult to read
- Difficult to maintain
- Difficult to reuse

Controllers solve this problem by separating business logic from routing.

---

# What is a Controller?

A Controller is a function that contains the **business logic** of an application.

It receives the request from the route, performs the required operations, and sends a response back to the client.

# Controller Syntax

```js
export const functionName = (req, res) => {

};
```

Example

```js
export const getBlogs = (req, res) => {

    res.status(200).json({
        message: "Blogs fetched successfully"
    });

};
```

---

# Responsibilities of a Controller

A controller is responsible for:

- Reading request data
- Validating input
- Calling the database
- Applying business logic
- Sending the response

---

# What Should NOT Be Inside a Controller?

Controllers should not contain:

- Route definitions
- Database connection code
- Application configuration

Each component should have a single responsibility.

---

# Route vs Controller

## Route

```js
router.get("/blogs", getBlogs);
```

The route only connects the URL to the controller.

---

## Controller

```js
export const getBlogs = async (req, res) => {

    const blogs = await Blog.find();

    return res.status(200).json({
        data: blogs
    });

};
```

The controller performs the actual work.

# Controller and Database

Controllers communicate with models.

Example

```js
const blogs = await Blog.find();
```

```js
const user = await User.create(data);
```

Controllers should never directly manipulate the database without using models.

---

# Controller and Middleware

Middleware prepares the request.

Example

```js
req.PrcUser = user;
```

Controller uses it.

```js
const userId = req.PrcUser._id;
```

# Common Mistakes

### Mistake 1

Writing business logic inside routes.

Wrong

```js
router.post("/register", async (req, res) => {

    // 100 lines of logic

});
```

Correct

```js
router.post("/register", registerUser);
```

---

### Mistake 2

Making controllers too large.

If a controller becomes very long, move reusable logic into utility functions or services.

---

### Mistake 3

Not returning after sending a response.

Wrong

```js
res.status(400).json({
    message: "Invalid"
});

console.log("Still running...");
```

Correct

```js
return res.status(400).json({
    message: "Invalid"
});
```

---

### Mistake 4

Connecting the database inside controllers.

Database connection should happen once when the server starts, not inside every controller.

---

# Best Practices

- Keep controllers focused on one responsibility.
- Write one controller for one operation.
- Keep routes clean.
- Validate incoming data.
- Return meaningful status codes.
- Use models for database operations.
- Keep controllers readable and organized.

---

# Summary

- A Controller contains the business logic of an application.
- Routes forward requests to controllers.
- Controllers process requests, interact with the database, and send responses.
- Controllers should not contain routing or server configuration.
- Separating controllers from routes makes applications easier to maintain, test, and scale.
