# Error First Callback

---

# Why

Many operations in Node.js take time to complete, such as:

- Reading files
- Writing files
- Connecting to databases
- Making network requests

Node.js does not wait for these operations to finish.

Instead, it executes them asynchronously.

To know whether an asynchronous operation succeeded or failed, Node.js introduced the **Error First Callback** pattern.

---

# What is an Error First Callback?

An Error First Callback is a callback function where the **first parameter always represents an error**.

If an error occurs, the first parameter contains the error object.

If no error occurs, the first parameter is `null`.

General Syntax

```js
function callback(error, data) {

}
```

---

# Structure

```js
(error, result)
```

- `error` → Contains the error if something goes wrong.
- `result` → Contains the successful result.

Only one of them will contain useful data.

---

# Successful Operation

```js
callback(null, data);
```

Meaning

- No error occurred.
- The requested data is available.

---

# Failed Operation

```js
callback(error, null);
```

Meaning

- An error occurred.
- No data is available.

---

# Example

```js
const fs = require("fs");

fs.readFile("demo.txt", "utf8", (error, data) => {

    if (error) {
        console.log(error);
        return;
    }

    console.log(data);

});
```

If the file exists:

```text
Hello World
```

If the file does not exist:

```text
Error: ENOENT...
```

---

# How It Works

```text
Start Operation

↓

Operation Completes

↓

Error?

↓

Yes ───────────────► callback(error, null)

↓

No

↓

callback(null, data)
```

---

# Why is Error First Important?

This pattern makes error handling consistent across Node.js.

Developers always know:

- First argument → Error
- Second argument → Result

This makes asynchronous code predictable.

---

# Node.js Built-in Modules

Many built-in Node.js modules use this pattern.

Examples

- File System (`fs`)
- Network
- Streams
- Timers

---

# Real Project Understanding

In modern Express applications, you rarely write Error First Callbacks directly.

Instead, libraries like:

- Mongoose
- Express
- JWT

return **Promises**, and we use:

```js
async
```

and

```js
await
```

instead.

Example

Instead of

```js
function(err, data) {

}
```

we now write

```js
try {

    const user = await User.find();

} catch (error) {

    console.log(error);

}
```

Modern JavaScript replaced most callback-based code with Promises and `async/await`.

---

# Callback vs Async/Await

## Callback Style

```js
fs.readFile("demo.txt", "utf8", (error, data) => {

    if (error) {
        console.log(error);
        return;
    }

    console.log(data);

});
```

---

## Async/Await Style

```js
try {

    const users = await User.find();

    console.log(users);

} catch (error) {

    console.log(error);

}
```

Async/Await is cleaner and easier to read.

---

# Callback Hell

Using many nested callbacks makes code difficult to read.

Example

```js
task1(function () {

    task2(function () {

        task3(function () {

            task4(function () {

            });

        });

    });

});
```

This is known as **Callback Hell**.

Promises and Async/Await solve this problem.

---

# Common Mistakes

### Mistake 1

Ignoring the error parameter.

Wrong

```js
fs.readFile("demo.txt", (error, data) => {

    console.log(data);

});
```

Always check for errors first.

---

### Mistake 2

Assuming both parameters contain values.

If `error` exists,

`data` is usually `null`.

If `data` exists,

`error` is `null`.

---

### Mistake 3

Using callbacks everywhere.

Modern backend development prefers:

- Promises
- Async/Await

Callbacks are mainly encountered in older Node.js APIs.

---

# Best Practices

- Always check the error parameter first.
- Return immediately after handling an error.
- Prefer `async/await` for modern applications.
- Understand callbacks because many Node.js APIs are built on this pattern.
- Avoid deeply nested callbacks.

---

# Summary

- Error First Callback is a standard Node.js pattern for handling asynchronous operations.
- The first parameter always represents an error.
- The second parameter contains the successful result.
- If no error occurs, the first parameter is `null`.
- Modern backend development mainly uses Promises and `async/await`, but understanding Error First Callbacks helps explain how Node.js evolved.
