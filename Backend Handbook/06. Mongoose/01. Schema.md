# Schema

---

# Why

MongoDB is a flexible database, meaning documents in the same collection can have different fields.

Example

Document 1

```json
{
    "username": "Kirat",
    "email": "kirat@gmail.com"
}
```

Document 2

```json
{
    "name": "Rahul",
    "phone": "9876543210"
}
```

This flexibility can lead to inconsistent data.

Mongoose solves this problem using **Schemas**.

A Schema defines the structure that every document should follow.

---

# What is a Schema?

A Schema is a blueprint or structure that defines:

- Fields
- Data Types
- Default Values
- Validation Rules
- Relationships

It tells Mongoose what a document should look like before storing it in MongoDB.

---

# Real World Analogy

Imagine a college admission form.

Every student must fill:

- Name
- Roll Number
- Email
- Course

The form decides:

- What fields exist
- What type of data is allowed
- Which fields are required

A Schema works in exactly the same way for database documents.

# Enum

Sometimes a field should only allow specific values.

Mongoose provides the `enum` option for this.

Example

```js
role: {

    type: String,

    enum: ["user", "admin"],

    default: "user"

}
```

Allowed Values

```text
user

admin
```

If someone tries to save

```text
manager
```

Mongoose throws a validation error.

Use `enum` when a field has a fixed list of allowed values.

Examples

- User Role
- Blog Status
- Payment Status
- Gender

---

# ref (Reference)

Sometimes one document needs to be connected to another document.

Instead of storing the entire document, MongoDB stores the **ObjectId** of the related document.

The `ref` property tells Mongoose which model that ObjectId belongs to.

Example

```js
authorId: {

    type: mongoose.Schema.Types.ObjectId,

    ref: "User",

    required: true

}
```

Here,

```text
authorId

↓

Stores ObjectId

↓

Points to User Collection
```

Example

```json
{
    "_id": "...",
    "title": "Learning Mongoose",
    "authorId": "687b7a23..."
}
```

The actual user information is stored in the **User** collection.

Later, we can use **populate()** to replace the ObjectId with the complete user document.

---

# Relationship Flow

```text
Blog Document

↓

authorId

↓

ObjectId

↓

User Collection

↓

User Document
```

`ref` only creates the relationship.

`populate()` uses that relationship to fetch the related document.

# Real Project Example

User Schema

```js
const userSchema = new mongoose.Schema({

    username: String,

    email: String

});
```

Blog Schema

```js
const blogSchema = new mongoose.Schema({

    title: String,

    description: String,

    authorId: {

        type: mongoose.Schema.Types.ObjectId,

        ref: "User"

    }

});
```

Every blog now belongs to one user.
Each field has a data type.

# Required Fields

A field can be marked as required.

Example

```js
username: {

    type: String,

    required: true

}
```

Now Mongoose will not allow saving a document without a username.

---

# Unique Fields

Prevent duplicate values.

Example

```js
email: {

    type: String,

    unique: true

}
```

Useful for:

- Email
- Username

---

# Default Values

Automatically assign a value if none is provided.

Example

```js
isAdmin: {

    type: Boolean,

    default: false

}
```

---

# Timestamps

Mongoose can automatically create:

- createdAt
- updatedAt

Example

```js
const userSchema = new mongoose.Schema(

    {

        username: String

    },

    {

        timestamps: true

    }

);
```

Result

```json
{
    "createdAt": "...",
    "updatedAt": "..."
}
```

---

# Complete Schema Example

```js
const userSchema = new mongoose.Schema(

    {

        username: {

            type: String,

            required: true

        },

        email: {

            type: String,

            required: true,

            unique: true

        },

        password: {

            type: String,

            required: true

        }

    },

    {

        timestamps: true

    }

);
```

# Schema Responsibilities

A Schema defines:

- Document structure
- Data types
- Required fields
- Default values
- Validation rules
- Relationships
- Timestamps

It **does not** communicate with MongoDB.

That is the job of the **Model**.

---

# Schema vs Model

| Schema | Model |
|---------|--------|
| Blueprint | Working Object |
| Defines Structure | Interacts with Database |
| No CRUD Operations | Performs CRUD Operations |
| One-time Definition | Used Throughout Application |

# Common Mistakes

### Mistake 1

Thinking Schema stores data.

❌ Wrong

A Schema only defines the structure.

The Model stores and retrieves data.

---

### Mistake 2

Confusing `type` with JavaScript types.

Wrong

```js
username: "String"
```

Correct

```js
username: String
```

---

### Mistake 3

Forgetting `timestamps`.

Without

```js
timestamps: true
```

MongoDB will not automatically create:

- createdAt
- updatedAt

---

### Mistake 4

Using `unique: true` as validation.

`unique` creates a unique index.

It does **not** replace proper validation or duplicate checks in your controller.

Example

```js
const existingUser = await User.findOne({
    email
});
```

---

# Best Practices

- Define clear field names.
- Use appropriate data types.
- Mark required fields.
- Use `timestamps: true`.
- Keep schemas focused on one collection.
- Add validation rules where necessary.

---

# Summary

- A Schema is the blueprint of a MongoDB document.
- It defines fields, data types, validation rules, and default values.
- Schemas help maintain consistent data.
- Schemas do not perform database operations.
- Models are created from Schemas and are responsible for interacting with MongoDB.
- Every collection in a Mongoose application should have its own Schema.
