# Validation

---

# Why

Users can send invalid or incomplete data.

Examples:

- Empty username
- Invalid email
- Negative age
- Very short password

If invalid data is stored in the database, the application becomes unreliable.

Mongoose Validation helps prevent invalid data from being saved.

---

# What is Validation?

Validation is the process of checking whether incoming data follows the rules defined in a Schema.

If the data violates a rule, Mongoose throws a validation error and the document is not saved.

# required

Makes a field mandatory.

Example

```js
username: {

    type: String,

    required: true

}
```

If no username is provided,

Mongoose throws a validation error.

---

# unique

Ensures duplicate values cannot exist.

Example

```js
email: {

    type: String,

    unique: true

}
```

Useful for

- Email
- Username

**Note:** `unique` creates a unique index. It is not a replacement for checking duplicates in your controller.

---

# minlength

Defines the minimum number of characters.

Example

```js
password: {

    type: String,

    minlength: 6

}
```

---

# maxlength

Defines the maximum number of characters.

Example

```js
username: {

    type: String,

    maxlength: 30

}
```

---

# trim

Removes extra spaces from the beginning and end of a string.

Example

```js
username: {

    type: String,

    trim: true

}
```

Input

```text
"   Kirat   "
```

Stored

```text
"Kirat"
```

---

# lowercase

Automatically converts text to lowercase.

Example

```js
email: {

    type: String,

    lowercase: true

}
```

Input

```text
KIRAT@GMAIL.COM
```

Stored

```text
kirat@gmail.com
```

---

# uppercase

Automatically converts text to uppercase.

Example

```js
countryCode: {

    type: String,

    uppercase: true

}
```

Input

```text
in
```

Stored

```text
IN
```

---

# enum

Allows only predefined values.

Example

```js
role: {

    type: String,

    enum: [

        "user",

        "admin"

    ]

}
```

Allowed

```text
user

admin
```

Not Allowed

```text
manager
```

---

# default

Provides a default value if none is supplied.

Example

```js
isVerified: {

    type: Boolean,

    default: false

}
```

---

# match

Validates data using a Regular Expression.

Example

```js
email: {

    type: String,

    match: /^\S+@\S+\.\S+$/

}
```

Useful for

- Email
- Phone Number
- ZIP Code

---

# Custom Validation

You can write your own validation function.

Example

```js
age: {

    type: Number,

    validate(value) {

        return value >= 18;

    }

}
```

Only ages 18 or above are accepted.

# Validation Example

```js
const userSchema = new mongoose.Schema({

    username: {

        type: String,

        required: true,

        trim: true,

        minlength: 3,

        maxlength: 30

    },

    email: {

        type: String,

        required: true,

        unique: true,

        lowercase: true

    },

    password: {

        type: String,

        required: true,

        minlength: 6

    },

    role: {

        type: String,

        enum: [

            "user",

            "admin"

        ],

        default: "user"

    }

});
```

# Schema Validation vs Controller Validation

Controller Validation

```js
if (!username || !email) {

    return res.status(400).json({

        message: "All fields are required"

    });

}
```

Purpose

- Better user experience
- Return custom error messages

---

Schema Validation

```js
required: true
```

Purpose

- Protect the database
- Prevent invalid documents from being stored

Professional applications use **both**.

# Common Mistakes

### Mistake 1

Thinking Schema Validation replaces Controller Validation.

Use both.

Controller validation improves user experience.

Schema validation protects the database.

---

### Mistake 2

Relying only on `unique`.

Always check for duplicates before creating a document.

Example

```js
const existingUser = await User.findOne({

    email

});
```

---

### Mistake 3

Forgetting `trim`.

Extra spaces can create inconsistent data.

---

### Mistake 4

Not validating user input.

Never trust data coming from the client.

---

# Best Practices

- Mark required fields.
- Use `trim` for text fields.
- Store emails in lowercase.
- Use `enum` for fixed values.
- Validate data in both controllers and schemas.
- Keep validation rules close to the Schema.

---

# Summary

- Validation ensures only valid data is stored in MongoDB.
- Mongoose provides built-in validators such as `required`, `unique`, `minlength`, `maxlength`, `trim`, `lowercase`, `uppercase`, `enum`, `default`, and `match`.
- Schema validation protects the database, while controller validation improves the user experience.
- Using both forms of validation results in secure, reliable, and production-ready applications.
